# **项目整体安排**

### 成员 A：项目管理 + LangGraph 核心图编排

**角色定位**：项目整体协调人，负责 ReAct 智能体的“大脑”——LangGraph 图结构设计与状态流转。

**具体任务**：

1. **定义全局状态对象**：编写 `State` 类（TypedDict），包含字段：
   - `user_query`：用户原始输入
   - `retrieved_papers`：已下载论文元数据列表
   - `current_chapter`：当前正在生成的章节
   - `outline`：已确认的综述大纲
   - `final_content`：已生成的内容字典（章节名→文本）
2. **构建 LangGraph 工作流图**：
   - 添加节点：`plan_outline`（规划大纲）、`search_papers`（调用检索工具）、`parse_and_index`（触发解析入库）、`generate_chapter`（调用生成工具）、`verify_references`（引用校验）、`finalize`（整合输出）。
   - 添加条件边：根据 `state["current_step"]` 决定下一节点，实现循环与分支。
3. **实现人机协同中断点**：
   - 在大纲生成后插入 `interrupt`，等待用户通过前端确认或修改提纲。
4. **工具封装与调用**：
   - 将成员 C 的检索函数、成员 B 的 RAG 检索函数、成员 E 的生成函数封装为 LangChain `@tool`，供 Agent 调用。
5. **编写图执行入口**：`run_survey_generation(user_input)` 函数，供成员 F 集成调用。

**协作接口**：

- 向成员 B 要求：`rag_search(query, top_k)` 函数签名。
- 向成员 C 要求：`search_arxiv(topic, max_results)` 返回标准化 JSON 列表。
- 向成员 E 要求：`generate_section(section_title, context_chunks)` 生成函数。

**产出物**：`graph.py`、`state.py`、`tools.py`（工具封装部分）。

------

### 成员 B：RAG 检索引擎 + Kotaemon 集成

**角色定位**：知识库的“心脏”，负责搭建高性能混合检索系统，确保每次生成都能获取最相关的论文片段。

**具体任务**：

1. **搭建 Kotaemon 基础环境**：

   - 安装并配置 Kotaemon，理解其 `Indexer` 和 `Retriever` 工作流。

2. **实现文档入库接口**：

   - 编写 `add_documents(chunks: List[Dict])` 函数，接收成员 D 处理好的文本块（包含文本、元数据、向量），调用 Kotaemon 入库。

3. **配置混合检索管线**：

   - 设置 BM25 全文检索（基于 Whoosh 或内存索引）。
   - 设置向量检索（使用 ChromaDB 或 Qdrant）。
   - 启用 Reranker（可选 Cohere 或本地 Cross-Encoder）提升精度。

4. **暴露统一检索接口**：

   - 实现 `retrieve(query: str, top_k: int = 5) -> List[Document]`，返回格式：

     python

     ```
     {
       "content": "段落文本",
       "metadata": {"paper_title": "...", "authors": "...", "year": 2023},
       "score": 0.87
     }
     ```

     

5. **编写检索质量测试脚本**：给定样例查询，人工验证返回结果的相关性。

6. **优化分块与召回参数**：根据实际效果调整 Chunk Size（建议 512 token）、Overlap（建议 50 token）、召回数量。

**协作接口**：

- 向成员 D 提供入库接口规范（Chunk 结构要求）。
- 向成员 A 提供 `retrieve()` 函数供其工具节点调用。

**产出物**：`rag_engine.py`、`config/kotaemon.yaml`（配置文件）、`retriever_test.py`。

### 成员 C：文档解析与向量化入库

**角色定位**：数据流水线的“消化系统”，负责将原始 PDF 转化为结构化的、可检索的文本块。

**具体任务**：

1. **PDF 文本提取与清洗**：
   - 使用 PyMuPDF (`fitz`) 逐页提取文本。
   - 编写启发式规则过滤页眉页脚（基于位置和字体大小）、剔除参考文献之后的冗余内容。
2. **论文结构识别**：
   - 利用正则表达式识别章节标题（如 `1. Introduction`、`2. Related Work`），为后续分块提供结构标记。
3. **语义分块器实现**：
   - 实现递归字符分割器，目标块大小 512 token（中文约 1024 字符，英文约 1500 字符）。
   - 保留与前一块 10% 的重叠，避免语义断裂。
4. **向量化与入库调用**：
   - 调用 Embedding API（初期可用 OpenAI `text-embedding-3-small` 或本地 `all-MiniLM-L6-v2`）将文本块转为向量。
   - 构造包含文本、向量、元数据（论文标题、作者、年份、所在章节）的数据结构。
   - 调用成员 B 提供的入库接口，将数据写入向量库。
5. **处理异常 PDF**：对于扫描版 PDF 或无文本层 PDF，输出警告日志并跳过。

**协作接口**：

- 从成员 C 获取下载好的 PDF 文件路径列表。
- 向成员 B 调用入库接口 `add_documents()`。

**产出物**：`pdf_parser.py`、`text_chunker.py`、`embedder.py`。

------

### 成员 D：**论文检索与 PDF 下载**

**角色定位**：数据入口的“采集器”，负责从学术数据库精准、可靠地获取论文元数据和全文。

**具体任务**：

1. **arXiv API 封装**：
   - 实现 `search_arxiv(query: str, max_results: int = 30, year_from: int = 2020)`。
   - 返回字段：`title`、`authors`、`summary`、`published`、`pdf_url`、`arxiv_id`。
2. **Semantic Scholar API 封装（补充源）**：
   - 实现 `search_s2(query, limit)`，用于获取 arXiv 覆盖不足的领域或获取引用量信息。
3. **结果去重与合并排序**：
   - 基于标题相似度（如 `fuzzywuzzy`）去重。
   - 按发表年份、引用量（若可获取）进行初排。
4. **异步下载管理器**：
   - 使用 `aiohttp` 或 `requests` + 多线程实现批量下载。
   - 实现重试机制（最多 3 次）、下载限速（间隔 1 秒）。
   - 将 PDF 保存至 `./data/pdfs/{arxiv_id}.pdf`，并返回本地路径列表。
5. **元数据持久化**：
   - 将论文元数据存入 SQLite 或 JSON 文件，供后续解析模块读取。

**协作接口**：

- 向成员 A 提供 `search_papers(topic, max_results)` 工具函数。
- 向成员 D 提供 PDF 存储路径及对应的元数据文件路径。

**产出物**：`paper_fetcher.py`、`downloader.py`、`metadata.db`（SQLite 结构）。

------

### 成员 E：**综述生成模块 + Prompt 工程**

**角色定位**：最终文本的“执笔人”，负责设计高质量的 Prompt，控制幻觉，确保综述内容学术规范。

**具体任务**：

1. **综述大纲生成 Prompt 设计**：
   - 输入：用户主题 + 已检索论文的标题/摘要聚类结果（可调用成员 A 传递的聚类信息）。
   - 输出：结构化大纲（JSON 格式），包含章节标题和子节要点。
2. **逐章生成 Prompt 设计**：
   - 系统指令：“你是一位学术综述作者，必须严格基于提供的参考资料撰写，不得添加外部知识。每个论断后需标注引用来源，格式为 `[1]`。”
   - 上下文构建：从成员 B 的检索结果中提取 Top-K 相关段落。
3. **引用对齐与幻觉校验**：
   - 编写正则提取生成文本中的引用标记 `[数字]`。
   - 回查向量库或元数据，验证该引用是否真实存在；若不存在则标记并剔除对应句子。
4. **实现章节生成函数**：
   - `generate_section(section_title, context_chunks) -> str`，供成员 A 的生成节点调用。
5. **内容后处理与整合**：
   - 将各章节拼接，生成目录，整理参考文献列表（按引用顺序排序）。
   - 提供 Markdown 格式的最终文稿。

**协作接口**：

- 从成员 B 获取检索结果（`retrieve()`）。
- 从成员 A 接收章节标题和上下文需求。

**产出物**：`prompts.py`（模板）、`generator.py`、`citation_checker.py`。

------

### 成员 F：**用户交互界面 + 系统集成**

**角色定位**：系统的“门面”与“粘合剂”，负责构建用户界面，并将所有模块串联为可运行的整体应用。

**具体任务**：

1. **搭建 Gradio Web 界面**：
   - 输入区域：主题文本框、时间范围滑块、文献数量选择、综述语言/格式下拉菜单。
   - 进度展示区域：实时日志框（显示“正在检索论文…”等状态）、进度条。
   - 结果展示区域：Markdown 预览组件、下载按钮（MD/PDF/Word）。
2. **实现前后端事件绑定**：
   - 将 Gradio 的按钮点击事件绑定到成员 A 的 `run_survey_generation` 函数。
   - 利用 Gradio 的 `gr.Progress` 和 `gr.Log` 实现进度反馈。
3. **多格式导出功能**：
   - Markdown：直接保存字符串为 `.md` 文件。
   - PDF：调用 `pandoc` 或 `reportlab` 将 Markdown 转换为 PDF。
   - Word：使用 `python-docx` 将 Markdown 内容写入 `.docx` 文件。
4. **系统联调与主流程脚本**：
   - 编写 `main.py` 或 `pipeline.py`，依次调用：C(检索下载) → D(解析入库) → A(启动 LangGraph，内部调用 B 和 E) → 返回最终文稿。
5. **编写用户文档与演示视频脚本**。

**协作接口**：

- 调用成员 A 的主入口函数。
- 调用其他成员模块时处理异常和错误提示。

**产出物**：`app.py`（Gradio 界面）、`exporter.py`、`pipeline.py`、用户操作手册。



任务的核心依赖链是：

**数据获取 (C) → 数据处理 (D) → 知识入库 (B) → 智能规划 (A) + 内容生成 (E) → 界面集成 (F)**

### Sprint 1 验收标准（Definition of Done）

1. **成员 C**：能通过命令行脚本传入关键词，成功下载至少 1 篇 arXiv PDF。
2. **成员 D**：能读取 PDF 并输出文本块列表及对应向量。
3. **成员 B**：能启动 ChromaDB，并将 D 的向量数据存入；能通过 `retrieve("test")` 返回存入的内容。
4. **成员 A**：`graph.py` 可运行并顺序打印节点名称（模拟流程）。
5. **成员 E**：产出至少 2 个版本的 Prompt 模板文档。
6. **成员 F**：Gradio 界面可启动，点击按钮后能触发后台模拟流程并在界面显示“完成”。
